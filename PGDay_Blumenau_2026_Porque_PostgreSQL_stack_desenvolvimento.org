#+LATEX: \clearpage
* Sobre esta palestra

- Antes de adicionar mais uma ferramenta à stack, vale perguntar se o PostgreSQL
  já resolve o problema com simplicidade suficiente.
- A ideia não é defender que tudo deve ser feito no PostgreSQL, mas trazer
  critérios para avaliar quando ele já entrega uma solução coerente, simples e
  operacionalmente viável.
- O que trago aqui é uma lista não exaustiva de funcionalidades importantes e
  onde elas podem ser aplicadas. 
- A pergunta não é: "dá para fazer no PostgreSQL?" mas "se o PostgreSQL resolve
  isso com simplicidade suficiente antes de eu assumir o custo de mais uma
  ferramenta?"

#+LATEX: \clearpage
* Observação sobre PostgreSQL 19

Os tópicos marcados como PostgreSQL 19 devem ser tratados como recursos de
versão em desenvolvimento até a publicação oficial da versão estável.

* 1. JSON / JSONB para dados semiestruturados
** Por que usar

Com JSON / JSONB...

- você armazena dados flexíveis sem abandonar o modelo relacional;
- fica mais fácil lidar com partes do domínio que têm estrutura variável, como
  eventos, configurações ou payloads de integração.

** Onde usar

- Eventos de aplicação
- Configurações por cliente
- Payloads recebidos de APIs externas
- Atributos opcionais de entidades

#+LATEX: \clearpage
** Exemplo 
#+begin_src sql
CREATE TABLE eventos (
  id bigserial PRIMARY KEY,
  payload jsonb NOT NULL
);

SELECT *
FROM eventos
WHERE payload @> '{"tipo": "checkout"}';
#+end_src

** Ponto de atenção

Se tudo vira JSONB...

- o modelo perde clareza;
- você perde constraints e tipagem explícita;
- a governança do modelo tende a piorar.

#+LATEX: \clearpage
* 2. Full Text Search (Busca textual nativa)
** Por que usar
Evita adicionar Elasticsearch ou OpenSearch cedo demais para buscas internas
simples ou médias.

** Onde usar
- Busca em artigos
- FAQ
- Catálogo interno
- Documentação
- Produtos ou posts

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
SELECT *
FROM posts
WHERE to_tsvector('portuguese', titulo || ' ' || corpo)
      @@ plainto_tsquery('portuguese', 'backup postgres');
#+end_src

** Ponto de atenção

Quando a busca começa a ficar mais sofisticada...

- relevância complexa pode justificar outro motor;
- autocomplete sofisticado pode justificar outro motor;
- análises de busca mais avançadas podem justificar outro motor.

#+LATEX: \clearpage
* 3. Índices GIN para busca, JSONB e arrays
** Por que usar
Acelera consultas em estruturas compostas, como documentos JSONB, arrays e
vetores de busca textual.

** Onde usar
- Full Text Search
- Filtros em JSONB
- Busca por tags
- Arrays consultados por contenção

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
CREATE INDEX idx_posts_busca
ON posts
USING GIN (to_tsvector('portuguese', titulo || ' ' || corpo));
#+end_src

** Ponto de atenção

Ao adotar índices GIN...

- o custo de escrita pode aumentar;
- o custo de manutenção pode aumentar;
- o consumo de espaço em disco pode crescer.

* 4. Arrays para listas simples
** Por que usar
Permite representar listas pequenas e simples sem criar tabelas auxiliares.

** Onde usar
- Tags simples
- Flags
- Pequenas listas de categorias
- Valores auxiliares de configuração

#+LATEX: \clearpage
** Exemplo

#+begin_src sql
CREATE TABLE artigos (
  id bigserial PRIMARY KEY,
  titulo text,
  tags text[]
);

SELECT *
FROM artigos
WHERE tags @> ARRAY['postgres', 'databases'];
#+end_src

** Ponto de atenção

Quando o relacionamento deixa de ser simples...

- relacionamentos importantes geralmente merecem modelagem normalizada;
- relacionamentos auditáveis geralmente merecem modelagem normalizada;
- relacionamentos muito consultados geralmente merecem modelagem normalizada.

* 5. Constraints como regras de integridade

** Por que usar

Com constraints...

- você mantém regras críticas próximas dos dados;
- diminui a chance de inconsistência causada por múltiplas aplicações acessando
  o mesmo banco.

** Onde usar
- Estados válidos
- Valores mínimos e máximos
- Unicidade
- Relacionamentos obrigatórios
- Regras de integridade de domínio

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
CREATE TABLE pedidos (
  id bigserial PRIMARY KEY,
  total numeric CHECK (total >= 0),
  status text CHECK (status IN ('novo', 'pago', 'cancelado'))
);
#+end_src

** Ponto de atenção

Constraints funcionam melhor quando...

- representam invariantes claras;
- a regra não muda o tempo todo na aplicação.

* 6. Views para encapsular consultas

** Por que usar

Com views...

- você padroniza consultas usadas por relatórios, APIs internas e módulos da
  aplicação;
- evita duplicar SQL em vários lugares.

** Onde usar
- Relatórios
- APIs de leitura
- Camadas de compatibilidade
- Modelos simplificados para BI

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
CREATE VIEW vw_pedidos_abertos AS
SELECT id, cliente_id, total, criado_em
FROM pedidos
WHERE status = 'aberto';
#+end_src

** Ponto de atenção

Quando as views começam a se empilhar...

- o custo pode ficar escondido;
- o troubleshooting pode ficar mais difícil;
- as dependências podem ficar frágeis.

#+LATEX: \clearpage
* 7. Materialized views como cache de leitura

** Por que usar

Com materialized views...

- você mantém resultados pré-calculados dentro do banco;
- adia a necessidade de adicionar outro componente de cache ou busca.

** Onde usar
- Dashboards
- Agregados por período
- Relatórios pesados
- Consultas recorrentes de leitura

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
CREATE MATERIALIZED VIEW mv_vendas_por_dia AS
SELECT date(created_at) AS dia, sum(total) AS total
FROM pedidos
GROUP BY 1;

REFRESH MATERIALIZED VIEW mv_vendas_por_dia;
#+end_src

** Ponto de atenção

Antes de tratar materialized view como cache...

- é preciso definir a estratégia de atualização;
- é preciso definir como lidar com concorrência;
- é preciso definir a tolerância a dados defasados.

#+LATEX: \clearpage
* 8. Triggers para automações próximas do dado

** Por que usar

Com triggers...

- você centraliza comportamentos transversais (aplicações diferentes);
- garante que eles aconteçam independentemente da aplicação que alterou os
  dados.

** Onde usar
- Auditoria
- Atualização de timestamps
- Histórico de mudanças
- Validações transacionais

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
CREATE FUNCTION set_updated_at()
RETURNS trigger AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_updated_at
BEFORE UPDATE ON clientes
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
#+end_src

** Ponto de atenção

Quando o comportamento fica escondido em triggers...

- para quem desenvolve, esse comportamento pode não ser óbvio;
- a manutenção pode ficar mais difícil.

#+LATEX: \clearpage
* 9. Functions e procedures para operações transacionais

** Por que usar

Com functions e procedures...

- você aproxima a lógica transacional dos dados;
- reduz múltiplas idas da aplicação ao banco.

** Onde usar
- Rotinas transacionais
- Processamento interno
- Cálculos reutilizados
- Operações críticas com atomicidade

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
CREATE FUNCTION cancelar_pedido(p_id bigint)
RETURNS void AS $$
BEGIN
  UPDATE pedidos
  SET status = 'cancelado'
  WHERE id = p_id AND status = 'aberto';
END;
$$ LANGUAGE plpgsql;
#+end_src

** Ponto de atenção

Quando a lógica começa a se concentrar demais no banco...

- um monólito escondido pode surgir;
- o versionamento pode ficar mais difícil.

* 10. LISTEN / NOTIFY para eventos leves

** Por que usar
Permite sinalização assíncrona simples entre sessões conectadas ao banco.

** Onde usar
- Invalidar cache
- Avisar workers
- Sinalizar novos eventos
- Atualizações em tempo quase real

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
NOTIFY novo_pedido, '{"pedido_id": 123}';

LISTEN novo_pedido;
#+end_src

** Ponto de atenção

Se o fluxo é crítico...

- não é uma fila durável;
- não substitui um broker.

#+LATEX: \clearpage
* 11. SKIP LOCKED para filas simples

** Por que usar
Permite que múltiplos workers consumam tarefas sem processar a mesma linha ao mesmo tempo.

** Onde usar
- Jobs internos
- Filas simples
- Processamento assíncrono leve
- Retentativas controladas pela aplicação

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
SELECT id
  FROM jobs
 WHERE status = 'pendente'
 ORDER BY id
   FOR UPDATE SKIP LOCKED
 LIMIT 1;
#+end_src

** Ponto de atenção

Quando a fila deixa de ser simples...

- quando o mesmo evento precisa ser consumido por vários processos (fanout), um broker
  pode fazer mais sentido;
- ordering forte pode exigir um broker;
- grande volume pode exigir um broker;
- políticas complexas de retry podem exigir um broker.

** Onde usar

* 13. Row Level Security para isolamento por linha

** Por que usar
Reforça segurança diretamente na camada de dados, especialmente em aplicações multi-tenant.

** Onde usar
- SaaS multi-tenant
- Dados por organização
- Dados por conta
- Ambientes com defesa em profundidade

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql

ON documentos
USING (cliente_id = current_setting('app.cliente_id')::int);
#+end_src

** Ponto de atenção

Se a RLS for mal configurada...

- ela pode criar uma falsa sensação de segurança;
- testes específicos passam a ser indispensáveis.

* 19. Generated columns para dados derivados

** Por que usar
Mantém cálculos derivados consistentes e próximos dos dados.

** Onde usar
- Campos calculados
- Normalização de busca
- Chaves derivadas
- Expressões reutilizadas

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
CREATE TABLE pessoas (
  nome text,
  sobrenome text,
  nome_completo text GENERATED ALWAYS AS (nome || ' ' || sobrenome) STORED
);

CREATE TABLE pessoas_virtual (
  nome text,
  sobrenome text,
  nome_completo text GENERATED ALWAYS AS (nome || ' ' || sobrenome) VIRTUAL
);
#+end_src

** Ponto de atenção

Quando a coluna é derivada...

- nem toda regra deve ir para o banco;
- a derivação precisa ser estável e clara.
- `STORED` grava o valor e calcula na escrita; `VIRTUAL` calcula na leitura;
- em PostgreSQL 18+, se você omitir `STORED` / `VIRTUAL`, o padrão é `VIRTUAL`;
- `VIRTUAL` tem mais restrições e funciona melhor em expressões simples com recursos nativos.

* 20. PostgreSQL 19: SQL/PGQ para consultas de grafo

** Por que usar
Permite expressar alguns problemas de relacionamento como padrões de grafo dentro do PostgreSQL.

** Onde usar
- Redes de relacionamento
- Dependências entre objetos
- Relações entre contas
- Hierarquias
- Recomendações simples

#+LATEX: \clearpage
** Exemplo 

#+begin_src sql
-- Exemplo conceitual:
-- usuarios(id, nome) e segue(perfil_origem_id, perfil_destino_id)

CREATE PROPERTY GRAPH social_graph
  VERTEX TABLES (
    usuarios KEY (id)
  )
  EDGE TABLES (
    segue
      KEY (perfil_origem_id, perfil_destino_id)
      SOURCE KEY (perfil_origem_id) REFERENCES usuarios (id)
      DESTINATION KEY (perfil_destino_id) REFERENCES usuarios (id)
  );

SELECT *
FROM GRAPH_TABLE (
  social_graph
  MATCH (u IS usuarios)-[s IS segue]->(v IS usuarios)
  COLUMNS (u.nome AS perfil_origem, v.nome AS perfil_destino)
);
#+end_src

** Ponto de atenção
- um banco de grafos especializado ainda pode fazer mais sentido quando o
  problema gira em torno de travessias complexas.

Com `UPDATE / DELETE FOR PORTION`...

- reduz lógica manual na aplicação.

** Onde usar
- Contratos
- Preços com vigência
- Assinaturas
- Políticas temporais
- Histórico de validade

#+LATEX: \clearpage

#+begin_src sql

Com `INSERT ON CONFLICT DO SELECT`...
